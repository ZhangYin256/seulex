# SeuLex 项目验收汇报稿

> 汇报时长：7-8 分钟

## 1. 项目背景与目标

这个项目的目标，是把 `.l` 词法规则文件编译成一个可运行的 C 扫描器。换句话说，用户只需要写词法规则、宏定义和动作代码，SeuLex 就能把它生成成标准的 `yylex()` 实现，后续可以和 Bison 生成的 parser 联调。

从代码上看，这个目标并不是简单的“文本替换”，而是一个完整的编译流水线：
- 先读入 `.l` 文件
- 再把规则中的宏展开成标准正则
- 解析正则表达式，生成 AST
- 基于 AST 构造 NFA
- 把 NFA 转成 DFA
- 对 DFA 做最小化
- 最后生成 C 代码

这样做的意义是：
1. 规则表达式不只是字符串，而是有明确语法和语义；
2. 自动机转换后，扫描速度更快；
3. 生成结果可直接编译，具备工程可用性；
4. 每一步都可以调试和验证，适合验收展示。

## 2. 总体实现流程（建议讲 2 分钟）

主入口在 `src/main.cpp`，它把整个流程串起来。可以按“输入—处理—输出”来讲。

### 2.1 输入阶段：读取配置和 `.l` 文件

程序先通过 `Config::parse(argc, argv)` 读取命令行参数，确定输入文件、输出文件、是否输出到 stdout、是否开启 debug。

随后 `LexInputScanner scanner(config.input_filename);` 读取 `.l` 文件并解析成 `SpecDocument`。这个结构里保存了：
- `macros`：宏定义表
- `rules`：词法规则列表
- `action_codes`：动作代码块
- `prologue` 和 `user_code_section`：前置代码和用户代码

这里的价值在于：SeuLex 不是只处理正则，而是处理整个 Lex 风格的规则文档。

### 2.2 正则预处理：宏展开

每条规则先经过 `FlexRegexConverter::convert()`。它会把 `{NAME}` 这样的宏引用展开成实际正则，并且会保留字符类 `[...]`、字符串字面量 `"..."` 和转义字符。

这一层的意义是避免“直接字符串替换”带来的优先级错误。比如如果宏内容本身包含 `|`，不加括号就会改变表达式语义，所以实现里会把展开结果包成 `(...)`。

### 2.3 正则解析：从字符串到 AST

`RegexParser` 负责把正则串解析成语法树。它支持的核心操作包括：
- 并列/连接 `ab`
- 选择 `a|b`
- 闭包 `*`
- 至少一次 `+`
- 可选 `?`
- 字符类 `[a-z]`
- 分组 `( ... )`
- 转义字符 `\n`、`\t`、`\x..`、八进制等

这个解析器采用的是典型的递归下降结构：
- `parse_alternation()` 处理 `|`
- `parse_concatenation()` 处理连接
- `parse_quantifier()` 处理 `* + ?`
- `parse_atom()` 处理最小单元

这种结构的好处是直观、易维护，而且和正则的优先级天然匹配。

### 2.4 自动机构造：AST → NFA → DFA → 最小化

这是整个项目最核心的一段。

先用 `NfaBuilder` 把 AST 编译成 NFA 片段。每种语法结构都对应一种 Thompson 构造：
- 字符 `a`：两个状态，中间一条字符边
- 连接 `ab`：前一个接受态通过 ε 边连到后一个起始态
- 选择 `a|b`：新增入口和出口，两边都可以走
- `*`：新增回路和空串路径
- `+`：和 `*` 类似，但至少走一次
- `?`：相当于“有或没有”两条路径

然后 `SubsetConstruction` 把 NFA 转成 DFA。核心算法是子集构造：
- 一个 DFA 状态对应一组 NFA 状态集合
- 先求 ε-closure（空转移闭包）
- 再对每个输入字符求 move
- 对 move 结果再求闭包，形成新的 DFA 状态
- 用 BFS / worklist 不断扩展直到没有新状态

最后 `DfaMinimizer` 对 DFA 做最小化。它先按“是否接受态 + 动作”做初始划分，再根据每个字符转移到的块编号继续细分，直到划分稳定。

### 2.5 代码生成：生成可运行 scanner

`ScannerGenerator` 输出的是完整的 C 代码，里面包含：
- 头文件
- 运行时缓冲
- 转移表 `transition_table[state][256]`
- 接受动作表 `accept_action[]`
- `yylex()` 主函数
- 可选 `main()` 测试入口

它的运行逻辑是：
1. 先把 stdin 全部读入缓冲区；
2. 从当前游标开始，用 DFA 尽量向前匹配；
3. 记录最后一个接受位置；
4. 匹配失败时向前跳过一个字符；
5. 匹配成功时根据接受状态执行对应 action。

这个设计能保证最长匹配，同时使生成器输出的代码可以直接编译运行。

## 3. 核心算法说明（建议讲 2.5-3 分钟）

### 3.1 正则解析算法

`RegexParser` 用的是递归下降。它不是一次性用正则库去解析，而是自己定义优先级：

1. `|` 最低优先级
2. 连接次之
3. `* + ?` 最高

例如表达式 `ab|c*` 会被解析成：
- 左边是连接 `a` 和 `b`
- 右边是 `c*`
- 最外层是选择 `|`

字符类处理也比较完整：
- 支持 `[^...]` 取反
- 支持区间 `a-z`
- 支持转义字符出现在字符类里

这保证了生成的 AST 是有语义层次的，而不是简单的字符串切分。

### 3.2 NFA 构造算法

这里用的是 Thompson 构造法，特点是：
- 构造简单
- 一定能从正则语法得到 NFA
- 对所有常见正则操作都容易实现

比如：
- `a|b`：用一个新起点，分别 ε 转移到 `a` 和 `b` 的 NFA，再从它们的接受态 ε 转移到新终点
- `ab`：把 `a` 的接受态 ε 连到 `b` 的起点
- `a*`：起点可以直接 ε 到终点，也可以进 `a`，且 `a` 的接受态可以回到自己起点形成闭环

这种 NFA 的优势是“构造容易”，但缺点是运行时需要处理 ε 转移，所以后面要转成 DFA。

### 3.3 子集构造算法

子集构造的思想是：把 NFA 的一组状态看成 DFA 的一个状态。

实现里有两个关键函数：
- `epsilon_closure(states)`：求闭包
- `move(states, c)`：在字符 `c` 下能到哪些状态

流程是：
1. 初始状态是 NFA 起点的 ε-closure；
2. 对每个 DFA 状态，遍历 256 个字符；
3. 用 `move` 找到下一组 NFA 状态；
4. 再做 ε-closure；
5. 如果这组状态以前没出现过，就创建一个新的 DFA 状态。

这样得到的 DFA 没有 ε 边，适合直接做高速扫描。

### 3.4 最长匹配和规则优先级

词法分析和普通字符串匹配不一样，它通常遵守两个原则：
1. **最长匹配**：能多吃一个字符就尽量多吃；
2. **规则优先级**：如果长度一样，先写的规则优先。

本项目在 DFA 状态上保留接受信息。扫描时：
- 一边前进一边记录最近一次接受的位置 `last_pos`
- 如果后续无法继续匹配，就回退到 `last_pos`
- 然后执行对应动作

同长度冲突时，`SubsetConstruction::resolve_accept_metadata()` 会挑 `rule_index` 更小的规则，这就实现了“先写的规则优先”。

### 3.5 DFA 最小化算法

`DfaMinimizer` 的思路是经典划分细化：

1. 先把状态分成两类：接受态和非接受态；
2. 接受态内部再按动作编号分组；
3. 比较每个状态在 256 个字符上的转移，看看是否会跳到不同的块；
4. 如果不同，就继续拆分；
5. 直到划分不再变化。

最终每个块代表一个最小 DFA 状态。

这个过程的好处是：减少状态数量，生成代码更紧凑，运行时表更小。

## 4. 项目特色与相较于常见实现的差异

### 4.1 不是“调用工具”，而是“自己实现编译链”

很多词法分析项目直接依赖 flex；本项目则把关键步骤自己实现了，包括宏展开、正则解析、自动机构造、最小化和代码生成。

这带来的好处是：
- 可以解释每一步为什么这么做；
- 可以展示内部结构；
- 方便做验收演示；
- 后续扩展更可控。

### 4.2 输出的是可联编的 C 扫描器

生成结果不是中间 IR，而是最终可编译的 C 文件，包含 `yylex()`、转移表、动作分发、`yytext` 和 `yyleng`。

这意味着它能直接接到 Bison 的 parser 上，完成端到端使用。

### 4.3 调试能力更强

`main.cpp` 在 `debug` 模式下能输出 NFA 的 Mermaid 图。这个功能在验收里很有用，因为你可以直接展示“为什么识别正确”，而不是只说结论。

### 4.4 语义处理更接近真实词法分析器

项目没有停留在“正则匹配”，而是考虑了：
- 规则优先级
- 动作代码嵌入
- 运行时缓冲
- 输出 `yylex` 接口

这说明它已经是词法分析器生成器，而不是单纯的正则引擎。

## 6. 可能被问到的细节问题与回答

### Q1：为什么不用直接模拟 NFA，而要转 DFA？
**A：**
NFA 模拟要处理大量 ε 转移和状态集合，运行成本高；DFA 把多个状态集合压缩成单一状态，扫描时只需要表查找，速度更稳定。

### Q2：为什么 DFA 还要最小化？
**A：**
最小化可以减少状态数，降低生成代码体积，也让转移表更小，提升缓存友好性。

### Q3：`RegexParser` 为什么自己写，而不是用库？
**A：**
因为要把正则直接变成可控的 AST，后面才能继续做 NFA 构造。库通常只给“匹配结果”，不给你结构化语法树。

### Q4：字符类 `[...]` 是怎么处理的？
**A：**
解析时会识别区间、转义和取反；构造 NFA 时会把字符类展开成 256 字节范围内的允许字符集合。

### Q5：宏展开为什么要检查循环引用？
**A：**
因为宏可以嵌套引用，如果 A 引用 B、B 又引用 A，会导致无限递归。检测到后直接报错，避免卡死。

### Q6：为什么生成器里用 `transition_table[state][256]`？
**A：**
因为输入是字节流，256 个字符可以直接索引数组，效率高，也能把运行时逻辑简化成一次查表。

### Q7：怎么保证“最长匹配”？
**A：**
扫描过程中不断更新最近一次接受位置，最后在无法继续时退回到该位置，而不是停在第一次接受的位置。

### Q8：规则顺序为什么重要？
**A：**
当多个规则匹配长度相同，词法分析通常按先出现的规则优先；本项目在接受态选择时保留了这个语义。

### Q9：生成的 scanner 为什么能和 Bison 配合？
**A：**
因为它输出的是标准 `yylex()` 接口，并且会把动作代码和用户代码嵌进最终 C 文件里，和 Bison 的 token 定义可以直接联编。

### Q10：当前实现还有什么可扩展点？
**A：**
可以继续补充起始条件、尾部上下文、更多 flex 兼容语法，以及更完善的错误恢复和测试覆盖。
